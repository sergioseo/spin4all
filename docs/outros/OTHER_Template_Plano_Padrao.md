# ⚡ Protocolo de Construção Antigravity (Spin4all)

Este documento define o **fluxo de execução mestre** para toda e qualquer interação de construção no projeto Spin4all. Ele transcende um simples checklist visual; é o nosso motor de alinhamento e criação de produtos de elite.

---

## 🛠️ Protocolo de Implementação (Padrão)

### 1. Leitura de Contexto (Obrigatório)
Antes de iniciar qualquer alteração em uma pasta:
- [ ] **Ler o `README.md`** da pasta alvo para entender o objetivo, linhagem e dependências.
- [ ] Verificar se a alteração impacta os elementos listados no documento.

### 2. Desenvolvimento com Documentação
Ao incluir novos itens, elementos ou dados:
- [ ] Documentar o novo elemento no `README.md` local imediatamente.
- [ ] Descrever a nova linhagem (entrada -> saída) se aplicável.

### 3. Finalização e Atualização
- [ ] Ao terminar, revisar a documentação da pasta para garantir que reflete o estado atual 100%.

---

## 🏗️ Fluxo de Trabalho

## 🧭 Fase 1: Discovery & Brainstorming (O "Entender")
Antes de qualquer linha de código, o Antigravity deve:
- [ ] **Categorizar a Ação**: É uma *Nova Construção*, *Refatoração*, *Ajuste de Lógica*, *Evolução de Design* ou *Manutenção*?
- [ ] **Mapear a Jornada**: Mostrar exatamente o que será feito em:
    - **Design & UI**: Quais componentes serão criados ou alterados.
    - **Backend & API**: Quais rotas e lógicas de servidor serão impactadas.
    - **Dados**: Alterações em banco ou seeds.
    - **Comunicação**: Como o sistema informará o usuário (loaders, modais, feedbacks).
- [ ] **Sondagem de Impacto e Legado**:
    - Identificar códigos antigos/conflitantes.
    - **Sugerir Melhorias**: O Antigravity deve usar sua versatilidade para propor caminhos mais eficientes ou criativos (o "Fator Uau" antecipado).

## 🛡️ Fase 2: O Portão de Alinhamento (O "Sinal Verde")
O Antigravity deve apresentar o plano e validar:
- [ ] **Entendimento 10/10**: O pedido foi compreendido em sua totalidade?
- [ ] **Alinhamento de Expectativa**: O resultado proposto atende ao que o usuário imaginou?
- [ ] **Aprovação Manual**: Se houver dúvida, **pare e pergunte**. Use recursos visuais (mockups/descrições) para garantir o alinhamento.

## 🏗️ Fase 3: Execução com Precisão e Cadência (O "Construir")
Durante a construção:
- [ ] **Consultar as Bases**: Seguir estritamente o `STYLE_GUIDE.md` e `CI_CD_METODOLOGIA.md`.
- [ ] **Construção Cadenciada**: Evoluir o código de forma organizada, sem pulos brutos que gerem erros escondidos.
- [ ] **Dúvida Zero**: Qualquer incerteza durante o código deve gerar uma nova pergunta ao usuário.

## 🔬 Fase 4: Guardião de Qualidade (O "Testar")
O Antigravity é seu próprio auditor crítico:
- [ ] **Teste no Navegador**: Validar funcionalidade e estética em tempo real.
- [ ] **Auto-Refatoração**: Se o resultado ficar "Nota 5", ele deve ser refeito/refinado internamente antes de ser apresentado.
- [ ] **Limpeza de Lixo**: Garantir que nenhum resquício de código morto ou conflituoso permaneça.

## ✅ Fase 5: Entrega e Validação (O "Validar")
- [ ] **Apresentação do Resultado**: Demonstrar a solução pronta.
- [ ] **Refinamento Final**: Ajustes finos baseados no feedback do usuário.
- [ ] **Deploy Autorizado**: Apenas após o sinal verde explícito.

---

### 🚀 O Mandato Antigravity
> *"Não estamos aqui apenas para seguir diretrizes; estamos aqui para entregar produtos que você nunca imaginaria ser capaz de construir sozinho."*

O Antigravity deve ser **veloz, versátil e criativo**. Estes padrões não são uma limitação, mas sim a base sólida para que a nossa comunicação seja fluida e o resultado seja sempre extraordinário.
